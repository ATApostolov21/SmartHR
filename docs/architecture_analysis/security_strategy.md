# Стратегия за Сигурност (Security Strategy Plan)

В момента проектът функционира като локално демо, където сигурността не е била приоритет. За да го превърнем в професионален, защитен продукт, трябва да изградим "крепостни стени" около него. Ето планът:

## 1. Аутентикация с JWT (Дигиталният Пропуск)

В момента всеки може да отвори Dashboard-а. Първата стъпка е да знаем **кой влезе**.

### Как работи (JSON Web Tokens)?
Представи си хотел с карти за достъп.
1.  **Login**: Потребителят въвежда "admin" и парола.
2.  **Токен (Картата)**: Сървърът издава `access_token` – дълъг низ от символи, подписан с таен печат.
3.  **Достъп**: Всеки път, когато браузърът иска данни ("Дай ми списъка със служители"), той показва този токен. Без токен = няма достъп.

### Имплементация:
*   Добавяме библиотека `python-jose` за създаване на токени.
*   Криптираме паролите с `bcrypt` (никога не пазим "123456" в текстов вид, а като неразбираема каша).

## 2. Rate Limiting (Защита от "Наводнение")

**Проблем**: Хакер (или развален скрипт) може да изпрати 1 милион заявки за секунда към нашия сървър (`/api/predict`), за да го блокира (DoS атака).

**Решение**: "Светофар" за заявки.
*   Използваме библиотека `slowapi`.
*   Правило: "Един IP адрес може да прави само 10 прогнози на минута".
*   Ако превиши: Сървърът връща грешка `429 Too Many Requests` ("Поизчакай малко").

## 3. Input Sanitization (Хигиена на Данните)

**Проблем**: Злонамерен потребител може да се опита да "инжектира" вирусен код.
*   *SQL Injection*: Тъй като не ползваме SQL, сме защитени от това.
*   *XSS (Cross-Site Scripting)*: В полето "Име на отдел" някой може да напише `<script>открадни_пароли()</script>`.

**Решение**:
*   Нашите **Pydantic** схеми (`api/schemas/employee.py`) вече действат като филтър. Те гарантират, че ако очакваме Число, ще получим Число, а не скрипт.
*   В Frontend-а React автоматично "ескейпва" (обезврежда) опасните символи, преди да ги покаже на екрана.

## 4. Управление на Тайните (Environment Variables)

**Грешката на начинаещия**: Да напишеш паролите директно в кода (`main.py`). Ако качиш кода в GitHub, целият свят вижда паролата ти.

**Правилният Начин**: `.env` файл.
*   Създаваме файл `.env`, който съдържа тайните:
    ```
    SECRET_KEY=супер_тайная_дълга_парола_за_токените
    ADMIN_PASSWORD=сложна_парола
    ```
*   Този файл **никога** не се качва в GitHub (слага се в `.gitignore`).
*   Python чете тези променливи само при стартиране.

## 5. RBAC (Role-Based Access Control)

Не всички служители са равни.
*   **Admin**: Може да качва файлове, да обучава модели и да вижда всичко.
*   **Analyst**: Може само да гледа графиките и да прави "What-If" симулации.
*   **Viewer**: Може само да гледа общия Dashboard.

**Имплементация**:
Във всеки JWT токен записваме ролята: `{"role": "analyst"}`.
Когато някой поиска "Изтрий файл", сървърът проверява: "Ти Admin ли си? Не? Достъп отказан."

---
**Обобщение**:
Сигурността не е една врата, а серия от проверки. Първо проверяваме кой си (Auth), после дали не прекаляваш (Rate Limit), дали носиш опасни неща (Sanitization) и накрая – дали имаш право да влезеш в тази стая (RBAC).
